# Changelog: 2026-05-19 — fix vendored RadEval imports after package rename

**Branch**: `debug-import` → PR into `hopprai/radeval:main`.
**Commit**: `caf0c4f fix(radgraph): rename vendored RadEval imports to radeval`

## Problem

After the `package-rename` PR (`260429.md`), users on Linux (case-sensitive
filesystem) received a `ModuleNotFoundError: No module named 'RadEval'` when
instantiating `RadEval(metrics=["radgraph", ...])`. The failure chain:

```
radeval/metrics/_radgraph_adapter.py
  → radeval/metrics/radgraph/__init__.py
    → from RadEval.metrics.radgraph._vendor.allennlp.data import Vocabulary
                                                                ↑ fails here
```

The `package-rename` PR updated `setup.py` and all non-vendored imports to
`radeval` (lowercase), but did not touch `radeval/metrics/radgraph/_vendor/` —
a frozen copy of the Stanford-AIMI/radgraph library. Every absolute import
in that tree still referenced the old capitalized name `RadEval`.

On case-insensitive filesystems (macOS, Windows) the rename resolved silently,
so the issue was not caught before release.

## Scope

742 import statements across 199 files, all inside `radeval/metrics/radgraph/_vendor/`:

| Location | Broken imports |
|---|---|
| `_vendor/allennlp/` | ~700+ |
| `_vendor/allennlp_models/` | ~30+ |
| `_vendor/dygie/` | ~12 |
| `_vendor/core.py` | 9 |

Broken imports included both top-level and indented deferred imports (inside
functions and `if TYPE_CHECKING:` blocks).

No non-vendored code was affected — outside `_vendor/`, `RadEval` appears
only as a class name, display string, logger name, or comment.

## Fix

Applied a cross-platform Python `re.MULTILINE` substitution scoped to
`from`/`import` lines (with optional leading whitespace):

```python
pattern = re.compile(r"^( *(?:from|import) )RadEval\b", re.MULTILINE)
```

This replaced every import-path reference to `RadEval` with `radeval`
while leaving comments, string literals, and class names untouched.

Three `RadEval` occurrences remain in `_vendor/` intentionally — all are
non-import prose:
- `_vendor/__init__.py:14` — docstring (`RadEval 2.1+ can run…`)
- `_vendor/core.py:45` — comment (`RadEval users who already have weights…`)
- `_vendor/allennlp/training/metrics/evalb_bracketing_scorer.py:105` — error
  message string (example CLI command)

## Verification

- `grep -rn 'from RadEval\|import RadEval' _vendor/` → 0 matches
- `python -m compileall _vendor/` → no syntax errors across all 199 files
- Smoke test: `from radeval.metrics.radgraph._vendor.allennlp.data import Vocabulary` → ok
- `tests/test_basics.py` → 1 passed
